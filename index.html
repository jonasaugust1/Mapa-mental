Resumo do Projeto

Descrição do produto

O app tem como funcionalidade principal colher informações e gerar um Relatório Operacional de voo que poderá ficar salvo offline e posteriormente ser mandado para a API

Publico Alvo

Empresas agrícolas que necessitam informatizar e fazer seus relatórios de maneira mais eficiente e rápida

Entidades

Defini as entidades de acordo com as informações necessárias para gerar o relatório. Essas entidades são objetos que servem para abstrair um modelo real para o código em que permitirá fazer as operações de CRUD

Relaciomentos

Relacionamentos são como as entidades interagem entre si
Um pra um 1-1
Um para muitos 1-n
Muito para muitos n-m

São eles:

    1-n

  • Cliente : Propriedade - 1-n ( Um cliente pode ter várias propriedades e uma propriedade só pode ter um cliente)
  • Propriedade : Pista - 1-n ( Uma propriedade pode ter várias pistas, mas uma pista está inserida em apenas uma propriedade)
  • Piloto : Relatório de Serviço - 1-n (Um piloto pode estar em vários relatórios, mas em um relatório somente pode haver um piloto)
  • Engenheiro : Relatório de Serviço - 1-n (Um engenheiro pode estar em mais de um relatório, mas em um relatório só pode haver um) engenheiro
  • Aeronave : Relatório de Serviço - 1-n (Uma aeronave pode estar em mais de um relatório, mas um relatório só pode conter uma aeronave)

    • n-m

  • Pista : Relatório de Serviço - n-m (Uma pista pode estar em vários relatórios e um relatório pode conter mais de uma pista)
  • Cliente : Relatório de Serviço - n-m (Um cliente pode estar em vários relatórios e em um relatório pode conter mais de um cliente)
  • Propriedade : Relatório de Serviço - n-m ( Uma propriedade pode estar em vários relatórios e em um relatório pode conter mais de uma propriedade)
  • Cultura : Relatório de Serviço - n-m ( Uma cultura pode estar em vários relatórios e um relatório pode conter mais de uma cultura)
  • Equipamento: : Relatório de Serviço - n-m ( Um equipamento pode estar em vários relatórios e um relatório pode conter mais de um equipamentos)

Conclusão

O projeto tem uma ampla utilidade e uma demanda considerável, além de ser útil e facilitar o trabalho de geração de relatórios.

    Passos a seguir

  • Definir um MVP do projeto para que possa ser feito e testado o mais rápido possível
  • Definir um banco de dados que encaixe melhor na estrutura do projeto
  • Definir as entidades
  • Criar a API observando seus relacionamentos, caso a escolha do banco de dados for um de modelo relacional
  • Criar as aplicações mobile, desktop e web
  • Fazer testes unitários, de integração e E2E
  • Conectar com a API
  • Fazer o deploy das aplicações e jogar para o cliente testar
